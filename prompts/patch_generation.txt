# 2. Diagnostic Analysis (诊断分析)

在生成任何补丁之前，请先根据 `Status` 进行深度诊断：
- **如果 Status 为 FAIL / INCORRECT**：仔细比对提取结果与标准答案。定位 Current Prompt 中哪一部分指令缺失、模糊或存在误导，从而导致了该错误。
- **如果 Status 为 PASS / CORRECT**：分析 Current Prompt 中是哪条具体的指令起到了关键作用，帮助模型做出了正确判断。**此时绝对不需要修改提示词！**

---

# 3. Modification Strategies (修改策略)

当你需要生成补丁时，请务必应用以下策略：

1. **Be Specific (指令具体化)**：
   - 拒绝废话：类似"提取时要小心"是无效指令。
   - 给出具体操作：类似"当表格有合并单元格时，先提取 row-span，然后填充每个单元格"是有效指令。
   - 始终提供具体的示例、准确的字段名称或明确的按步操作指南。

2. **Match Specificity to Failure Frequency (匹配错误频率)**：
   - 偶发错误：使用宽松的文本说明（在末尾追加一条规则）。
   - 频发错误：使用极其严格的规则限制，例如强制检查清单或设立"DO NOT（禁止）"黑名单。

3. **Preserve What Works (保留有效内容)**：
   - 如果某个案例通过了测试，定位起作用的指令，**绝对不要修改或删除它**。
   - 只能增强或保持不变，绝不移除产生正确结果的指引。

4. **Improve Conciseness (提升简洁度)**：
   - 如果当前提示词已经有冗余表述，优先整合而不是继续叠加。
   - 用精简的规则或一针见血的例子代替长篇大论。

5. **Tighten on Recurring Errors (拦截高频错误)**：
   - 若评估持续误判某一特定模式，请提供明确的决策树或检查清单。
   - 使用"当...时，切勿（DO NOT）判定为 PASS"等强否定词。

6. **Align to Ground Truth (强行对齐真值)**：
   - 深入分析真值背后的业务合理性，生成的补丁必须使用户能稳定复现真值结果。

# 4. Patch Operations & Rules (补丁操作与规范)

根据分析结果生成 JSON 数组格式的补丁。优先使用小粒度的 patch（修改几行），切忌用大 patch 重写整个章节。

## 支持的操作列表

| op (操作指令) | 必需字段 | 说明 |
|---|---|---|
| `append_to_section` | `target_section`, `content` | 在章节末尾追加内容。**（最高优先级，最推荐使用）** |
| `insert_after` | `target_section`, `target_text`, `content` | 在指定文本之后插入。 |
| `insert_before` | `target_section`, `target_text`, `content` | 在指定文本之前插入。 |
| `replace_in_section` | `target_section`, `old_text`, `new_text` | 替换章节中的文本。**（⚠️高风险：`old_text` 必须与原提示词一字不差精确匹配，否则将失败！）** |
| `replace_section` | `target_section`, `content` | 完全重写整个章节内容（会保留原有 header）。 |
| `add_after_section` | `target_section`, `new_header`, `content` | 在目标章节之后新增一个全新章节。 |
| `delete_section` | `target_section` | 删除整个章节。 |

## ⚠️ 严格约束 (CRITICAL RULES)
1. **成功即静默**：如果执行状态是 `PASS` 或 `CORRECT`，**必须且只能**输出空数组 `[]`，严禁生成任何无意义的补丁。
2. **操作偏置**：新增规则时必须优先使用 `append_to_section`。极力避免使用 `replace_in_section`，除非你必须修改已有文字（且能保证逐字复制原句）。
3. **保护区限制**：如果目标 section 标记有 `[PROTECTED]`，则严禁对其进行内部编辑，只能使用 `replace_section` 进行整体替换。
4. **精确 target_section**：`target_section` 字段必须与 Prompt Structure 中列出的 section ID **一字不差**。

---

# 5. Output Format (输出格式)

请输出一个合法的 JSON 对象，包含 `patches` 和 `cited_sections` 两个字段。不要输出任何额外的解释文本。

```json
{{
  "patches": [
    {{
      "op": "append_to_section",
      "target_section": "section_id_here",
      "content": "新增的具体判断规则以修复某类误判",
      "reasoning": "由于模型在提取XX时漏掉了YY，此处补充说明"
    }}
  ],
  "cited_sections": [
    "section_id_1",
    "section_id_2"
  ]
}}
```

**字段说明**：

* `patches`: 本次需要执行的修改操作集合。（**注意：状态为 PASS/CORRECT 时，此处必须为空数组**）
* `cited_sections`: 本次分析时参考了原提示词中的哪些 section（从 Prompt Structure 中选择精准的 section ID）。若为错误样本，填入导致错误的 section；若为正确样本，填入促成正确的关键 section。
