# Merge Guidelines
请在合并过程中严格遵守以下三大维度的核心指标：

### 一、 结构划分与隔离原则
* **GROUP BY SECTION（分类隔离）**：首先按 `target_section` 对补丁进行归类。不同 section 的补丁绝对不得混淆（如 Rules 的归 Rules，Output_Format 的归 Output_Format）。
* **ISOLATION（异类隔离）**：若某个 patch 在其 section 内解决的是完全独特的边界情况（孤例），必须完整保留其修改意图，**切勿**为了追求精简而强行删减。
* **保序性**：同一 section 内的 `append` 或 `insert` 操作，必须按照系统底层的业务逻辑合理顺序进行排列。
* **ICL 保护**：严禁修改带有 `[PROTECTED]` 标记的 section 内部结构。

### 二、 逻辑去重与泛化抽象
* **去重与拼接**：同一 section 内针对相同意图的多条 patch 必须合并。同 section 且同 op 的多条 patch：`append` 的 content 直接进行文本拼接，`replace` 的 `new_text` 进行融合。
* **泛化抽象（同类归纳）**：若同一 section 内多个 patch 在解决"同一类相似问题"，需提取共性模式（Prevalent-pattern），将个例级指令抽象为通用原则。*（例如：将 "img1.jpg 的表格有合并单元格" 抽象为 "当表格存在合并单元格时..."）*
* **冲突消解**：当同一 section 内出现相互矛盾的操作时，优先选择理由（reasoning）更充分的一方，或将其调和为更完善的独立操作。
* **流行度偏置**：优先采纳在多个独立 patch 中反复出现的修改（代表系统性漏洞）；仅在单个 patch 中出现的孤立修改需要更强的理由支持才予保留。

### 三、 技术约束与硬性底线（防崩溃）
* **独立性（行级互不重叠）**：**【核心硬性约束】** 合并后的每一条 edit 在行级上必须互不重叠。任意两条 edit 绝不能修改同一行或同一段文本（因为 Edits 将被并行应用，行冲突会导致系统应用失败）。
* **操作偏置（优先 Append）**：新增规则时必须优先使用 `append_to_section`。仅在必须修改/纠正已有文字时才允许使用 `replace_in_section`，且其 `old_text` 必须与原 Prompt 全文**逐字精确匹配**。
* **精简压缩（软性目标）**：合并后的总条数原则上应显著少于输入总量（目标 ≤ 原始数量的 1/3），但该目标**不得**以牺牲"异类隔离"的有效补丁为代价。

# Output Format
请直接输出合并后的 JSON 数组，**不要添加任何 markdown 标记包裹之外的解释性文字、开场白或总结**。

```json
[
  {{
    "op": "append_to_section",
    "target_section": "## Section Name",
    "content": "合并并提炼后的新增内容",
    "reasoning": "合并了 X 个关于 xxx 的相似 patch，提取共性进行了通用原则抽象"
  }},
  {{
    "op": "replace_in_section",
    "target_section": "## Section Name",
    "old_text": "Prompt中原有的精确文本（必须逐字复制，严禁臆造）",
    "new_text": "替换后的全新文本内容",
    "reasoning": "冲突消解与合并说明：替换了原有的 xxx 错误陈述"
  }}
]
```
