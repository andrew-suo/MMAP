# Merge Workflow

请严格按以下顺序合并：

1. **Group by section**：按 `target_section` 分组。不同 section 的 patch 不得混合。
2. **Deduplicate**：同 section、同意图、同边界条件的 patch 必须合并，保留最具体、最可执行、措辞最清晰的版本。
3. **Specific-to-general**：多个 patch 暴露同类失败模式时，提炼成一条通用规则，而不是保留多个样本级规则。
4. **Preserve unique insights**：解决不同边界情况的 patch 必须保留其独特意图，不得为了压缩而删除。
5. **Resolve conflicts**：若 patch 之间相互矛盾，选择 reasoning 更强的一方，或合成为一条更完整且不冲突的规则。
6. **Enforce line independence**：合并后的任意两条 edit 不得修改同一行、同一段文本或同一 `old_text`。如果多条新增规则会落到同一 section，优先合并为一条 `append_to_section`。

# Merge Guidelines

## 一、结构隔离
- `target_section` 必须来自 Prompt Structure 中已有的 section id。
- 带 `[PROTECTED]` 标记的 section 不允许内部编辑；如确需修改，只能使用整体替换类操作。
- 同一 section 内的新增规则应按判断/执行的自然顺序排列。

## 二、去重、泛化与保留
- 高频重复 patch 表示系统性漏洞，应合并为更通用的规则。
- 单例 patch 只有在 reasoning 说明清楚且边界风险明确时保留。
- 合并时可以精炼 `content`，但不得改变原 patch 的修复意图。
- 不要输出空泛规则，例如“更仔细地检查”“注意准确性”。
- `source_patch_ids`、来源记账和 lineage 由外部程序统一绑定；不要臆造、猜测或依赖它们来完成合并决策。

## 三、技术硬约束
- 优先输出 `append_to_section`。
- 只有当必须修正已有文字，且 `old_text` 能从当前 prompt 精确复制时，才允许 `replace_in_section`。
- 禁止两个输出 patch 使用相同或重叠的 `old_text` / `target_text`。
- 输出 patch 总数应小于等于输入中独特意图的数量；不要为了数量目标删除独特有效补丁。

    "reasoning": "说明该 patch 是保留、去重、泛化、冲突消解中的哪一种，以及覆盖了哪些原始意图"
  }},

    "op": "replace_in_section",

    "new_text": "替换后的文本",
    "reasoning": "说明为何必须替换已有文字，以及如何避免与其他 edit 重叠"
  }}
